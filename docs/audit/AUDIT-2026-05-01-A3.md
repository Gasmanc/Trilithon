## 4. Internal consistency findings

### 4.1 Slice continuity — PASS

Every slice file declares its Entry conditions explicitly. Spot checks across phases 4, 9, 13, 18, and 25 (the phases with the largest slice counts) confirm that slice N.k+1 either references slice N.k as a precondition or is explicitly noted as independent of it. No forward references to slices that come later in the same phase were found.

### 4.2 Cross-phase dependencies — PASS

Every cross-phase reference cites a prior phase by number and a concrete artefact (a trait method, a SQL table, a typed mutation, a frontend component). Spot checks against the phased plan confirm dependent phases are sequenced before their consumers. Notably:

- Phase 7 (apply path) correctly waits for Phase 4 (mutation algebra) and Phase 5 (snapshot writer).
- Phase 11 (web UI shell) correctly waits for Phase 9 (HTTP API).
- Phase 17 (concurrency control) correctly references Phase 4's `expected_version` envelope and Phase 9's 409 response, consistent with the Stage A reframing applied during the prior round.
- Phase 27 (Tier 2 hardening) cites every Tier 2 feature it exercises end-to-end.

One soft note: Phase 11 slice 11.5 expects `apply_in_flight` on `/api/v1/health` and slice 11.8 expects `POST /api/v1/desired-state/validate`. Neither endpoint is currently scheduled in Phase 9. Phase 11 explicitly flags this as an open question (see §5 below).

### 4.3 Audit `kind` discipline — PARTIAL PASS

Architecture §6.6 declares 43 audit kinds and asserts the regex `^[a-z][a-z0-9-]*(\.[a-z][a-z0-9-]*)+$` is enforced at the storage adapter boundary. Cross-reference of dotted tokens used in slice files identifies eight new kinds the slice corpus implies but the §6.6 vocabulary does not yet list. They are flagged in §6 below for ratification. No slice silently invented a kind; every divergent kind was raised in the slice file's "Open questions" subsection or carried a `### Audit kinds emitted` line citing the unratified name.

### 4.4 Tracing event discipline — PARTIAL PASS

Architecture §12.1 declares 36 tracing event names and span field keys. Cross-reference identifies four new event names the slice corpus implies but the §12.1 table does not yet list. These are flagged in §6 below for ratification.

A subtle drift exists between §6.6 and §12.1 naming conventions on the same logical event:

- §6.6 (audit kind): `caddy.capability-probe-completed` (hyphenated final word).
- §12.1 (tracing event): `caddy.capability-probe.completed` (dotted final word).

Both forms are valid under their respective regexes. The architecture should declare the divergence is intentional (audit kinds carry the action verb in a single hyphenated noun, tracing events split the verb out as a separate dot-segment). Recommendation in §10 below.

### 4.5 Trait reference discipline — PASS

All 11 traits declared in `trait-signatures.md` are referenced by at least one slice, with reference counts: `Storage` 116, `CaddyClient` 46, `Applier` 24, `ProbeAdapter` 20, `DiffEngine` 19, `EnvProvider` 19, `SecretsVault` 15, `ToolGateway` 12, `HttpServer` 8, `DockerWatcher` 8, `PresetRegistry` 2.

The two `PresetRegistry` references both come from Phase 18 (the only phase that uses it), which is correct. The low count is not an issue.

Two trait extensions are surfaced by the slice corpus (see §6 reconciliation table 3): `Storage::upsert_tls_certificates`, `list_tls_certificates`, `upsert_upstream_health`, `list_upstream_health` (Phase 14), and `ToolGatewayError::ValidationFailed` (Phase 20).

### 4.6 Path discipline — PARTIAL PASS

Slice files use both `crates/<name>/...` (workspace-relative, anchored at `core/`) and `core/crates/<name>/...` (repo-rooted) forms. Per the README §"Path conventions" both are acceptable, but the canonical form is the workspace-relative one. The mixed usage does not block implementation; it does mean a less-capable agent may need to resolve the prefix once per slice.

The `crates/ffi` directory exists in the workspace scaffold but is not used by any V1 slice. One slice (Phase 1's pre-flight checklist) acknowledges its existence as a Phase 0 artefact and does not extend it. This is correct.

## 5. Open questions roll-up

The five authoring agents collectively surfaced 30 open questions. Aggregated, deduplicated, and classified below. Each entry carries a recommended resolution; the project owner may override.

**Classification.** HARD = blocks implementation of at least one slice; resolution required before the relevant slice begins. SOFT = a default exists or can be picked by the implementer with low risk of rework.

### 5.1 HARD — Resolve before relevant phase begins

**H1. Audit kind for ownership-sentinel takeover (Phase 3).** The `--takeover` flag writes an `OwnershipSentinelTakeover` audit row but §6.6 only lists `caddy.ownership-sentinel-conflict`. Recommendation: ADD `caddy.ownership-sentinel-takeover` to §6.6 (see §6 reconciliation). The two events are semantically distinct (a conflict is an observation, a takeover is an action), so a dedicated kind is correct.

**H2. Audit kinds for restore (Phase 26).** `RestoreApplied` and `RestoreCrossMachine` event names have no §6.6 counterpart. Recommendation: ADD `system.restore-applied` and `system.restore-cross-machine` (see §6 reconciliation). The `system.*` namespace is new but consistent with `secrets.*`, `auth.*`, and the other §6.6 namespaces.

**H3. Audit kind for Caddyfile import (Phase 13).** §6.6 lists `config.imported`; Phase 13 slices emit `import.caddyfile`. The Stage A reconciliation already added `import.caddyfile` to §6.6 with `config.imported` retained as legacy alias. **Recommendation: this is already resolved**, but the implementer SHOULD verify §6.6 still reflects this when starting Phase 13.

**H4. Tracing event `access-logs.capacity-90-percent` (Phase 22).** §12.1 lacks this event. Recommendation: ADD to §12.1. Convention: `<area>.<observation>` (consistent with `storage.integrity-check.failed`).

**H5. Tracing event `drift.skipped` (Phase 8).** Phase 8 slice 8.5 fires this when drift detection is suppressed because Caddy is unreachable. §12.1 does not list it. Recommendation: ADD to §12.1.

**H6. Tracing events `secrets.file-backend.startup` and `secrets.master-key.permissions-tightened` (Phase 10).** Phase 10 slices 10.4 fires these. §12.1 does not list them. Recommendation: ADD both to §12.1.

**H7. Trait extension `ToolGatewayError::ValidationFailed` (Phase 20).** Phase 20 slice 20.3 introduces a fifth variant on the closed `ToolGatewayError` enum. Recommendation: ADD `ValidationFailed { errors: ValidationErrorSet }` to `trait-signatures.md` §7. The current four-variant enum cannot represent propose-mode validation failure cleanly.

**H8. Trait extension `Storage` for TLS and upstream tables (Phase 14).** Phase 14 slice 14.1 adds four new methods (`upsert_tls_certificates`, `list_tls_certificates`, `upsert_upstream_health`, `list_upstream_health`) to the closed `Storage` trait. The `tls_certificates` and `upstream_health` SQL tables also need migration files. Recommendation: ADD the four methods to `trait-signatures.md` §1; ADD migrations (numbered consistent with prior migrations) for `tls_certificates` and `upstream_health` tables; UPDATE architecture §6 data model with the table DDL.

### 5.2 SOFT — Resolution can be deferred to implementation

**S1. Phase 2 `mutations` table name (mutations vs mutations_queue).** Architecture §6.7 names it `mutations`; the Phase 2 reference text uses `mutations_queue`. The slice ships `mutations` (architecture wins). Recommendation: RATIFY architecture's name; update Phase 2 reference text to match. No code change required.

**S2. Phase 2 initial-migration table set.** The Phase 2 reference lists seven tables in `0001_init.sql`, omitting tables defined in architecture §6 (`drift_events`, `proposals`, `tokens`, `policy_presets`, `route_policy_attachments`, `capability_probe_results`, `routes`). Recommendation: keep the seven tables in `0001_init.sql` and add the others in subsequent migrations as their consuming phases land. The slice corpus already follows this pattern.

**S3. `ShutdownSignal` placement (Phase 1 → Phase 2).** Group A recommended a trait-based `ShutdownObserver` in `core` with a `tokio::watch` impl in `adapters`. Recommendation: ACCEPT this. The trait pattern preserves the §5 layer rule (no tokio in core).

**S4. `apply_mutation` rollback purity (Phase 4 → Phase 7).** A `Rollback { target: SnapshotId }` mutation cannot resolve a snapshot id without storage access, so `apply_mutation` cannot be pure for that variant. Recommendation: ACCEPT a pre-resolved `RollbackContext` parameter pre-populated by the caller via `Storage::get_snapshot`. Document the asymmetry on the doc comment for `apply_mutation`.

**S5. Snapshot column drift (Phase 5).** Phase 5 reference adds `created_at_monotonic_nanos` and `canonical_json_version` columns; architecture §6.5 does not yet document them. Recommendation: ADD both columns to architecture §6.5 in the same commit that lands the migration.

**S6. Migration number collision (Phase 5).** Phase 5 reference names `0002_snapshots_immutable.sql`; Phase 3 already consumes `0002_capability_probe.sql`. The slice ships `0004_snapshots_immutable.sql`. Recommendation: ACCEPT slice's renumbering. Phase 5 reference SHOULD adopt the new number.

**S7. Phase 6 vocabulary gaps for AuditEvent enum (mappings).** The phase reference uses event names that map to §6.6 dotted strings via the `Display` impl. The slice corpus uses the §6.6 form. Recommendation: ACCEPT slice form. Phase 6's reference SHOULD adopt the dotted strings as the canonical names; the Rust `AuditEvent` enum variant names are an implementation detail.

**S8. Bootstrap-creation audit row absence (Phase 9).** Phase 9 slice 9.4 leaves the bootstrap-creation event without an audit row, observable only via the 0600 file mode. Recommendation: ADD an audit row at first-run bootstrap creation. Reuse `auth.bootstrap-credentials-rotated` with a `notes.creation = true` field, or add `auth.bootstrap-credentials-created`. Recommended: the latter (cleaner kind disambiguation).

**S9. Slice 7.4 ApplyError vs ApplyOutcome::Failed split.** The slice maps Caddy 4xx to `Ok(ApplyOutcome::Failed)` (typed soft-failure) and reserves `Err(ApplyError)` for transport- and programmer-level conditions. Recommendation: ACCEPT this split. The semantic is "soft failure was an expected outcome; hard failure was unexpected." Document the rule on the `Applier` trait doc comment.

**S10. Mutation audit-row count for happy path (Phase 9).** The slice emits `mutation.applied` from the HTTP layer plus `config.applied` from the applier. Recommendation: ACCEPT both. They describe different events at different layers (acceptance of the mutation request vs successful reconciliation with Caddy). Document this in architecture §7.1.

**S11. Phase 10 redactor hash semantics (per-row vs per-plaintext).** Slice picks per-row. Recommendation: ACCEPT. XChaCha20 nonces guarantee distinct ciphertexts even for identical plaintext; per-row hashing is consistent with that.

**S12. Caddy version skew between Phase 13 and Phase 16.** Phase 13 round-trip pins Caddy 2.11.2; Phase 16 demo pins Caddy 2.8 (the floor). Recommendation: KEEP both pins. Phase 16 demo runs against the floor for compatibility verification; Phase 13 tests against current stable for forward compatibility. This was an intentional decision in Stage A; reaffirm it in `deploy/compose/UPGRADING.md`.

## 8. Cross-cutting risks

**8.1 Vocabulary ratification debt.** Eight of the 30 open questions are vocabulary additions the slice corpus needs to ship in the same commit that begins the relevant phase. If the implementer forgets, the storage adapter's regex check (and the architecture §12.1 closed-table rule) will reject the new entries. Recommendation: when starting any of the affected phases (3, 8, 10, 14, 20, 22, 26), the first slice should land the §6.6 / §12.1 / `trait-signatures.md` updates as a precondition to all other slices in that phase.

**8.2 Cross-phase API discovery.** Phase 11 surfaces three endpoints (`apply_in_flight` on `/api/v1/health`, the routes-list view, `POST /api/v1/desired-state/validate`) that Phase 9 does not yet schedule. Recommendation: add three slices to Phase 9 (or fold into existing Phase 9 slices) that ship these endpoints before Phase 11 begins. Alternatively, surface them as Phase 15 work (the validate endpoint specifically belongs there).

**8.3 Path-form mixing.** The `crates/...` and `core/crates/...` forms both appear. Recommendation: declare the workspace-relative form (`crates/...`) canonical in the README and run a one-off sweep through the slice corpus. This is cosmetic and does not block implementation.

**8.4 Phase 5 word count below the soft floor.** Phase 5 came in at 3,058 words against a 3,000-word floor. Six of seven slices are dense Rust signatures and SQL DDL, which is the desired depth. The slice file does not need padding.

**8.5 Phases 8, 20, 22 below the 4,000-word soft floor.** These three slice files are each at 2,947 / 3,280 / 3,737 words. The depth bar is met (full signatures, named tests, concrete acceptance commands), so the under-target word counts reflect concise content rather than missing content. No action recommended.
