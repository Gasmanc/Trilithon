## 2. Executive summary

The specification set is unusually thorough and internally well-cross-referenced. Phase 13 (Caddyfile import), Phase 17 (concurrency control), Phase 18 (policy presets), Phase 23 (Docker Compose deployment), Phase 24 (systemd deployment), and Phase 25 (configuration export) are at or near the "almost wrote the code" bar across all seven dimensions. The Tier-1 phases that establish the durable plumbing (Phase 1 daemon skeleton, Phase 5 snapshot writer, Phase 6 audit log, Phase 7 apply path) are tight on module paths and acceptance commands but thinner on full Rust signatures and full data shapes than later phases.

The two systemic weaknesses are, in order of severity:

- **Audit `kind` vocabulary drift.** The architecture document at §6.6 publishes a closed vocabulary of audit `kind` values and an enforcement regex. Several phase TODOs introduce new `kind` values (`import.caddyfile`, `export.caddy-json`, `export.caddyfile`, `export.bundle`, `mutation.rebased.auto`, `mutation.rebased.manual`, `mutation.rebase.expired`, `mutation.rejected.missing-expected-version`, `docker.socket-trust-grant`, `policy.registry-mismatch`) that the §6.6 table does not list. A less capable agent will not know whether to extend §6.6 or to use the closest existing kind. This must be reconciled before implementation begins.
- **Configuration key drift between phases for the Caddy admin endpoint.** The canonical key is `[caddy] admin_endpoint` (architecture §8.1, ADR-0010, ADR-0011, ADR-0015). Phase 24's `config.toml.example` task names the key `[caddy] admin_socket`. The two MUST agree on one name.

A small number of phases miss the threshold on one or two dimensions but are within striking distance: Phase 1, Phase 4, Phase 8, Phase 11 (frontend signatures less precise than later UI phases), Phase 14, and Phase 15. None of the 27 phases is fundamentally unfit for handoff; the gaps are narrow and addressable in a single edit pass per phase.

## 3. Per-phase scorecards

### Phase 1 — Daemon skeleton (17/21)

D1: 2  D2: 2  D3: 2  D4: 2  D5: 3  D6: 3  D7: 3.

Gaps:

- **D2 (type signatures).** The TODO names types (`DaemonConfig`, `ExitCode`, `ConfigError`) but does not give the full Rust signature for the loader. The current text states only: "`crates/adapters/src/config_loader.rs` MUST read a TOML file from a documented path, overlay environment variables prefixed `TRILITHON_`, and return either `DaemonConfig` or a typed `ConfigError`." Recommend: specify the function signature explicitly, for example `pub fn load_config(path: &Path, env: &dyn EnvProvider) -> Result<DaemonConfig, ConfigError>`, and enumerate the `ConfigError` variants in the same place the enum is introduced (`Missing { path }`, `MalformedToml { line }`, `EnvOverride { var, reason }`, `DataDirNotWritable { path }`).
- **D3 (data shapes).** The `DaemonConfig` struct is enumerated by field name only, not by Rust type. Recommend: state the field types: `listen_address: IpAddr`, `listen_port: u16`, `data_dir: PathBuf`, `caddy_admin_endpoint: CaddyEndpoint` (where `CaddyEndpoint` is `enum { Unix(PathBuf), LoopbackTls { url: Url, mtls_cert_path: PathBuf, mtls_key_path: PathBuf } }`), `drift_check_interval_seconds: u32`, `tracing_log_filter: String`, `bootstrap_mode: bool`. This puts the disambiguation between `unix:///run/caddy/admin.sock` and `https://127.0.0.1:2019` in the type, where it belongs.
- **D1 (module paths).** The TODO names paths like `crates/cli/src/cli.rs` and `crates/core/src/exit.rs` (correct relative form), but a less capable agent might also see the architecture say `core/crates/cli/...` and become confused about whether the prefix `core/` is implicit. Recommend: state once at the top of every TODO that all crate paths are rooted under `core/` in the workspace, so `crates/core/src/foo.rs` resolves to `core/crates/core/src/foo.rs` on disk. The same recommendation applies to every other phase that uses the bare `crates/...` form.

### Phase 4 — Mutation algebra (17/21)

D1: 2  D2: 2  D3: 2  D4: 2  D5: 3  D6: 3  D7: 3.

Gaps:

- **D2 (type signatures).** The TODO says "Implement `DesiredState::apply_mutation`. Acceptance: `apply_mutation(state, mutation, capabilities) -> Result<MutationOutcome, MutationError>` MUST be pure and side-effect-free." That is a near-signature, but it elides ownership and lifetimes. Recommend: write `pub fn apply_mutation(state: &DesiredState, mutation: &Mutation, capabilities: &CapabilitySet) -> Result<MutationOutcome, MutationError>` and add a doc comment example. The `MutationError` enum variants are also not enumerated: list them inline (`Validation { rule, path, hint }`, `CapabilityMissing { module, required_by }`, `Conflict { observed_version, expected_version }`, `Schema { field, kind }`) consistent with the architecture's `core::Error` variants in §4.1.
- **D3 (data shapes).** The TODO lists `Route`, `Upstream`, `Host`, etc. by name only. The architecture also names them but no document states their full Rust structure. Recommend: specify each record's field set inline in this TODO (so the implementer does not have to derive them from Caddy documentation): for `Route`, `id: RouteId, hostnames: Vec<HostPattern>, upstreams: Vec<UpstreamId>, matchers: MatcherSet, headers: HeaderRules, redirects: Option<RedirectRule>, policy_attachment: Option<PolicyAttachment>, enabled: bool`.
- **D1 (module paths).** The TODO says `crates/core/src/model/` (a directory). For a less capable agent, name the per-file split: `model/route.rs`, `model/upstream.rs`, `model/policy.rs`, `model/matcher.rs`, `model/header.rs`, `model/redirect.rs`, `model/secret.rs`, `model/mod.rs`.
- **D4 (algorithms).** Capability gating is described as "rejected at validation with a typed `MutationError::CapabilityMissing`". Recommend: specify the algorithm as numbered pseudocode: (1) extract the set of Caddy modules referenced by the mutation; (2) intersect with `capabilities.loaded_modules`; (3) for each non-empty residue, push `MutationError::CapabilityMissing { module, required_by: mutation.kind() }`; (4) if the residue list is non-empty return Err early.

### Phase 8 — Drift detection (17/21)

D1: 2  D2: 2  D3: 2  D4: 2  D5: 3  D6: 3  D7: 3.

Gaps:

- **D4 (algorithms).** The structural diff engine is described as "compare two `DesiredState` values structurally and produce a typed `Diff` value usable by audit, drift detection, and rollback preflight." This is high-level. Recommend: spell the diff algorithm out: (1) flatten each `DesiredState` to a sorted `BTreeMap<JsonPointer, JsonValue>`; (2) compute the symmetric difference; (3) classify each path as `Added`, `Removed`, or `Modified { before, after }`; (4) ignore the documented set of fields Caddy is allowed to mutate (TLS issuance state, upstream health caches, the `automatic_https.disable_redirects` field). The architecture document mentions the ignore list at §7.2 but the TODO does not point at it; cite §7.2 and reproduce the list inline.
- **D2 (type signatures).** `DriftEvent` is described by field set but not as a Rust struct. Recommend: write `pub struct DriftEvent { before_snapshot_id: SnapshotId, running_state_hash: String, diff_summary: HashMap<ObjectKind, DiffCounts>, detected_at: UnixSeconds }` with `pub struct DiffCounts { added: u32, removed: u32, modified: u32 }`.

## 5. Cross-cutting risks

Five patterns appear across multiple phases.

**5.1 Rust trait method signatures elided to method names.** Phases 2, 3, 4, 6, 7, 8, 10, 12, 14 all describe trait surfaces by enumerating method names rather than full async signatures with ownership and error types. The architecture document does not paper over this either (section 4 names types but not method signatures). The fix is mechanical (one edit per trait), but a less capable agent will produce non-uniform signatures. Recommendation: a single appendix at the end of the architecture document or an addendum at `docs/architecture/trait-signatures.md` listing every workspace trait with its full Rust signature. Each phase TODO then cross-references that appendix instead of restating signatures.

**5.2 Frontend component signatures inconsistent across phases.** Phases 17, 18, 25 give explicit React component signatures (`export function PolicyTab(props: { routeId: string }): JSX.Element`). Phases 11, 14, 15 do not. The pattern Phase 17 set is the right one. Recommendation: backfill Phase 11, 14, 15 to use the Phase 17 form. This also keeps the "Co-locate styles via Tailwind classes" rule from the project CLAUDE.md visible at the per-component level.

**5.3 Tracing span attribute names not specified anywhere.** The architecture document at §12 says correlation identifiers propagate through `tracing::Span`, and individual phases mention emitting tracing events with named attributes (`daemon.started`, `storage.migrations.applied`, `storage.integrity_check.failed`, `docker.socket-trust-grant`). No document publishes the closed set of tracing event names or the closed set of span field names. A less capable agent will invent these inconsistently. Recommendation: add a tracing vocabulary table to the architecture observability section, mirroring the audit `kind` vocabulary at §6.6. The contents are largely already named across the phases; collecting them in one place is the work.

**5.4 Test-file naming convention not stated.** Phase 17 names integration test files under `core/crates/adapters/tests/concurrency_*.rs`. Phase 18 names them `core/crates/adapters/tests/policy_*.rs`. Phase 25 names them `core/crates/adapters/tests/export_<format>.rs`. Phase 13 uses `core/crates/adapters/tests/caddyfile_round_trip.rs`. The pattern is `<area>_<scenario>.rs` but the rule is not stated. Recommendation: add a one-paragraph naming convention at the top of `docs/todo/README.md` and reference it from the phased plan.

**5.5 The `expected_version` envelope appears in Phase 5 (snapshot writer) but is not formally introduced until Phase 17 (concurrency control surface).** Phase 5 mentions it in the `Snapshot` record. Phase 4's `Mutation` enum does not declare it on the variants. Phase 9's HTTP API surface introduces a 409 conflict response but does not require `expected_version` on every request. Phase 17 retroactively requires `expected_version` on every typed-mutation entry. A less capable agent implementing Phase 4, 5, 7, 9 sequentially will not know to thread `expected_version` through the request envelopes from the start. Recommendation: state in Phase 4 (mutation algebra) that every variant carries `expected_version: i64`, and state in Phase 9 that every mutation handler accepts `expected_version` in the request body, even though the conflict-handling UI does not arrive until Phase 17. The Phase 17 work then becomes "surface the existing 409 to the user," not "retrofit `expected_version` everywhere."

1. **Audit `kind` vocabulary authority.** Is the §6.6 table the authoritative source, or do the phase TODOs override it? Recommend declaring §6.6 authoritative and updating it to include every kind any phase emits.
