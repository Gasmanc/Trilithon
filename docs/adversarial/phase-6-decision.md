# Design Decision — Phase 6

**Date:** 2026-05-08
**Rounds:** 24
**Final approach:** Phase 6 ships a tamper-evident, append-only audit log built around an `AuditWriter` that serialises all writes through a single `tokio::sync::Mutex<Option<SqliteConnection>>` with `BEGIN IMMEDIATE` locking. Each row carries a `prev_hash = sha256(canonical_json(predecessor))` chain verified at startup via a stateful batch API (`ChainVerifyState` / `verify_batch` / `verify_finish`). Secrets are gated by a `RedactedDiff` newtype (constructed only through `SecretsRedactor`) whose `v1:` format is locked in for this phase; `SecretsRevealed` events are blocked until Phase 10. The `AuditEvent` enum uses a single manual `impl fmt::Display` (no `strum::Display`) with an `Unknown(String)` catch-all variant to maintain forward-hash-stability with future phases; `PredecessorRow` with an explicit 16-field primitive layout mediates all sqlx DB reads to preserve the `RedactedDiff` opacity invariant.

## Rejected Approaches

| Approach | Rejected because |
|----------|-----------------|
| Omitting `prev_hash` column from the schema entirely | R1-F001: ADR-0009 requires a per-row tamper-evidence chain; the entire tamper-detection guarantee was absent with no test to catch it |
| Using a plain `String` for `redacted_diff_json` rather than a `RedactedDiff` newtype | R1-F002: without a compile-time type boundary, any call site can store plaintext secrets; the newtype makes bypass a compile error |
| Deriving the redaction hash from encrypted-at-rest ciphertext in Phase 6 | R1-F003: Phase 10 vault does not exist yet; using ciphertext hash creates an irreconcilable format split; resolved by adopting `secret:v1:<sha256-of-plaintext>` |
| Migration numbered `0003_audit_immutable.sql` | R1-F004: `0003_seed_local_instance.sql` already occupied that prefix; renamed to `0006_audit_immutable.sql` |
| Deriving `correlation_id` from ambient `Span::current()` inside `record`, removing it from `AuditRow` | R2-F103: every non-request call site (startup, background tasks, tests) would silently receive a random synthetic ID with no compile-time enforcement; `correlation_id` kept as a required field on `AuditRow` |
| Using a `Transaction<'_, Sqlite>` shared between the business write and the audit write for atomicity | R3-F201 / R4-F204: SAVEPOINT does not serialise across concurrent WAL connections; sqlx's internal depth counter is corrupted by raw SAVEPOINT SQL; replaced by a dedicated `Mutex<SqliteConnection>` with `BEGIN IMMEDIATE` |
| Using `DEFAULT ''` on `prev_hash` with a `chain::verify` skip for `prev_hash == ""` | R3-F203: backfill populates `0000…0000` not `""`; the skip-on-empty logic becomes a permanent bypass for write-path bugs producing empty hashes; skip condition changed to match the ZERO_SENTINEL value |
| Running `ALTER TABLE … ADD CHECK` as migration 0007 to enforce `occurred_at = occurred_at_ms / 1000` | R3-F202: SQLite `ALTER TABLE` does not support `ADD CONSTRAINT`; migration would fail at deployment; replaced by an application-layer guard and a startup consistency query |
| Using a single `chain::verify(rows: &[AuditRow])` that pre-loads all rows into memory | R2-F108 / R20-M2: would OOM or time out on large deployments; replaced by a stateful batch API (`ChainVerifyState` / `verify_batch` / `verify_finish`) with 500-row pages |
| Hashing only `prev_hash` of the predecessor (a chain of hashes of hashes) rather than `sha256(canonical_json(predecessor))` | R13-H1: step 5 originally fetched only `SELECT prev_hash … LIMIT 1`, making `canonical_json(predecessor)` impossible; this revealed the design had two incompatible interpretations; step 5 now mandates a full named-column projection to allow `canonical_json` computation |
| Leaving `Storage::record_audit_event` as a surviving parallel write path after introducing `AuditWriter` | R4-F301: any call site still using the old path inserts `prev_hash = ''`, permanently breaking `chain::verify`; `record_audit_event` was explicitly retired with a CI grep recipe |

| Using `#[strum(to_string = "{0}")]` on `AuditEvent::Unknown` to make `to_string()` return the inner string | R18-H1: strum treats `to_string` attribute values as static string literals, not format templates; every unrecognized kind would hash as the literal `"{0}"`; replaced by the manual `Display` impl with an `Unknown(s) => write!(f, "{}", s)` arm |
| `#[derive(strum::Display)]` alongside `strum::EnumString` as the Display strategy | R19-H1: identical compile error to R15-H2; the vocabulary consolidation task now explicitly prohibits `#[derive(strum::Display)]` |
| Using `std::sync::Mutex<Option<SqliteConnection>>` for the `AuditWriter` connection slot | R20-M1: `std::sync::MutexGuard` is `!Send`; holding it across `.await` points in an `async fn` raises `error[E0277]` on any multi-thread Tokio executor; replaced by `tokio::sync::Mutex` |
| `chain::verify(rows: impl Iterator<Item = &AuditRow>)` as a synchronous iterator API called from an async paginator | R20-M2: no idiomatic bridge exists between async sqlx I/O and a synchronous iterator without either pre-loading all rows (OOM risk) or blocking a Tokio thread; replaced by the stateful `verify_batch` / `verify_finish` API |

## Key Constraints Surfaced

The adversarial process revealed these constraints that any implementation must respect:

1. **All audit writes must be serialised through a single `tokio::sync::Mutex<Option<SqliteConnection>>` opened with `BEGIN IMMEDIATE`** — concurrent pool connections in WAL mode can both read the same predecessor hash and produce a duplicate chain link; this is the only correct serialisation mechanism without giving up business-transaction atomicity for audit writes.

2. **`prev_hash` must be bound from the DB-computed value (`new_prev_hash`), never from `row.prev_hash`** — `AuditRow.prev_hash` as a caller-supplied field is silently ignored at write time; binding the struct field would immediately and permanently break the chain from the first write (R12-H1).

3. **`canonical_json` must use the 17 DB column names as JSON keys (e.g., `"redacted_diff_json"`, not `"redacted_diff"`), derive `occurred_at` as `occurred_at_ms / 1000` internally, include `prev_hash`, represent NULLs as JSON null, sort keys lexicographically, and emit no whitespace** — any deviation causes permanent false `ChainBroken` reports on immutable rows.

4. **`AuditEvent::Unknown(String)` must be the forward-compatibility catch-all for unrecognised `kind` strings**, and both `record` step 5 and the startup paginator must apply identical fallback logic; the `to_string()` of `Unknown(s)` must return `s` unchanged (manual `Display` arm required, not strum derive).

5. **`Actor::from_kind_id` unknown-kind fallback must produce `System { component: format!("__unknown:{}", kind) }` at both `record` step 5 and the startup paginator** — differing fallback strings between the two call sites cause the write-time hash and the verify-time hash to diverge permanently (R16-H1).

6. **The startup paginator must use `PredecessorRow` (an explicit 16-field primitive struct) to retrieve DB rows** — `sqlx::FromRow` cannot map `Option<RedactedDiff>` or `Actor` directly; implementing `sqlx::Type<Sqlite>` for either type violates the `RedactedDiff` opacity invariant; `PredecessorRow` uses `Option<String>` for `redacted_diff_json` and separate `String` fields for `actor_kind`/`actor_id`, with conversion via `RedactedDiff::from_db_str`.

7. **Migration 0006 must follow the sequence: `ALTER TABLE` to add `prev_hash`, then `UPDATE` to backfill `ZERO_SENTINEL`, then `CREATE TRIGGER`** — if triggers are installed before the backfill `UPDATE`, the immutability trigger blocks the backfill; the entire migration is atomic via sqlx's per-file transaction.

8. **`SecretsRevealed` events are blocked at `record` until Phase 10** — `PHASE6_REGISTRY` contains placeholder paths that do not match real `DesiredState` diff fields; unredacted secrets would be written to the immutable log if this variant were permitted (R3-F208, R4-F303).

9. **`verify_batch` must use per-row `prev_hash` inspection** (not a "have we seen a non-sentinel yet" flag) to classify each row as empty-hash error / ZERO_SENTINEL / chained; the flag-based approach diverges on out-of-order data and produces different `last_computed_hash` states.

10. **The dedicated audit connection must set `PRAGMA busy_timeout = 5000` and `PRAGMA foreign_keys = ON`** on both initial open and all recovery reopens; `busy_timeout` is required to avoid indefinite Mutex hold time under write contention (R8-F801); `foreign_keys` enforces the `snapshot_id REFERENCES snapshots(id)` FK that SQLite disables by default (R17-L1).

11. **`occurred_at` must never appear as a field on `AuditRow`** — it is derived from `occurred_at_ms / 1000` at bind time and inside `canonical_json`; if the field were present it could be set from the raw DB column and diverge from the derivation, corrupting chain hashes for pre-Phase-6 rows (R16-M2).

12. **The `AuditWriter` connection slot must use `tokio::sync::Mutex`, not `std::sync::Mutex`** — `std::sync::MutexGuard` is `!Send` and cannot be held across `.await` in a multi-thread Tokio executor (R20-M1).
